# 任务卡 Slice s03 - 会话状态机闭环与单场景真实结算

## 1. 本卡要交付的用户能力
用户在 `RUNNING/SETTLING/DONE` 会话中可完成开始、结束、重试、返回场景选择，并在结束时看到基于真实几何评估的成功/失败结果（含 `0.05` 与 `0.051` 速度阈值差异）。

## 2. 本卡为什么现在做
- 解决核心缺口：当前结束逻辑仍是 `successHint` 占位，缺少真实结算与可回合化会话控制。
- 消灭最大风险：若不先打通 `结束 -> 结算 -> 展示` 主链路，后续风险策略（碰撞/压线/超时）无法可靠落地。
- 对应 first playable 阶段：从“可控可动”推进到“可完成一局并得到可信结果”。

## 3. 触达的闭环链路
- Trigger: 用户首次有效控制开始会话；用户点击结束；用户点击重试/返回场景选择。
- Input: 控制输入（`direction/throttle`）、结束动作、当前场景几何、自车姿态与速度信息。
- Runtime/State: `READY -> RUNNING -> SETTLING -> DONE`，并支持 `DONE -> READY/IDLE` 回流。
- Core Logic: `session-controller -> session-end-policy -> result-classifier -> result-panel`。
- Derived Facts / Settlement: 产出结算结果（成功/失败）与关键判定事实（含速度阈值边界）。
- UI Output: HUD、控制区、结果面板同步显示阶段与结算结果，可执行重试或返回。

## 4. 影响的状态机
- 当前涉及状态：`READY`、`RUNNING`、`SETTLING`、`DONE`（含返回场景后的 `IDLE/READY`）。
- 新增/修改的状态迁移：
- 有效控制触发 `READY -> RUNNING`（保持既有逻辑）。
- 用户点击结束触发 `RUNNING -> SETTLING`。
- 结算完成触发 `SETTLING -> DONE`。
- 用户点击重试触发 `DONE -> READY`（保留场景并重置会话）。
- 用户返回场景选择触发 `DONE -> IDLE`（清空当前会话上下文）。
- `DONE` 下控制输入默认不生效，直到重试或重新选场景。
- 不在本卡处理的状态：碰撞/压线/超时自动结束与风险锁存（切片4处理）。

## 5. 影响的数据流
- 输入数据：结束动作、重试动作、返回动作、会话中的控制输入、当前场景标识与几何信息。
- 运行时状态：`phase`、`selectedScenario`、`lastControl`、`ego`、结算中间态。
- 派生事实：结算结果、失败原因（如速度超阈值）、边界值判定（`0.05` 成功，`0.051` 失败）。
- 展示状态：结果面板文案、HUD 状态、控制区按钮可用性。

## 6. 范围（In Scope）
- 实现会话“开始/结束/重试/返回场景选择”状态机闭环。
- 接通 `session-end-policy` 与 `result-classifier` 的真实结算链路（单场景）。
- 将 `0.05` 与 `0.051` 的速度阈值差异纳入结算逻辑并可观察。
- 在结果面板展示真实结算结果并与会话状态一致。

## 7. 非范围（Out of Scope）
- 不处理碰撞/压线/超时/输入越界风险策略（切片4）。
- 不扩展四场景循环与场景差异完整验收（切片5）。
- 不做无关 UI 重构与视觉风格升级。

## 8. 实现边界 / 约束
- 不允许修改 `auth/`、`infra/`。
- 不允许新增生产依赖。
- 公共接口若发生变化，必须补 contract test。
- 不引入临时 debug hook 到生产路径。
- 保持既有图层顺序与渲染契约不被破坏，遵守仓库门禁（`lint/test/build`）。

## 10. 验收标准（行为式）
1. Given 场景已加载且会话处于 `RUNNING`，When 用户点击“结束”，Then 会话进入 `SETTLING` 后到 `DONE`，结果面板显示真实结算结果。
2. Given 会话满足成功条件且结束瞬时速度为 `0.05`，When 触发结算，Then 结果为成功。
3. Given 其他条件不变但结束瞬时速度为 `0.051`，When 触发结算，Then 结果为失败且显示速度超阈值信息。
4. Given 会话已 `DONE`，When 用户点击“重试”，Then 会话回到 `READY` 且自车姿态重置；When 点击“返回场景选择”，Then 退出当前会话上下文。
5. Given 会话已 `DONE` 且未重试，When 用户继续输入控制，Then 会话状态与车辆姿态保持不变。

## 11. 测试策略（有预算）
- 单元测试：`result-classifier` 的阈值判定与结果分类分支。
- 状态机测试：`session-controller` 的 `RUNNING -> SETTLING -> DONE` 及 `DONE -> READY/IDLE` 迁移。
- 集成 smoke：
- “加载场景 -> 控制 -> 结束 -> 结果展示”主链路。
- “结束后重试再结束”回合链路。
- 本卡不测：碰撞/压线/超时锁存、四场景差异完备性。

## 12. 实施步骤（建议 3~5 步）

- Step 2：在 `session-end-policy` 中接入分类器，形成结束策略编排。

- Step 5：补充 normal/boundary/smoke 测试并执行仓库门禁验证。

## 13. 风险与回滚
- 主要风险：阈值边界浮点误差导致 `0.05/0.051` 判定不稳定；状态迁移与 UI 显示可能出现时序不一致。
- 回滚边界：仅回滚 `session-controller`、`session-end-policy`、`result-classifier` 与对应 UI 接线，不触碰渲染基础契约。
- 无破坏确认：确保既有 s01/s02 渲染与输入行为测试继续通过，再确认 s03 新增验收项通过。

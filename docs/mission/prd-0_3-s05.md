# 任务卡 Slice s05 - 四场景循环练习与可区分渲染完整闭环

## 1. 本卡要交付的用户能力
用户在 `READY/DONE` 状态切换四个场景后，都能在画布看到可区分的场景渲染，并可完成控制、结束与结果结算闭环。

## 2. 本卡为什么现在做
- 解决核心缺口：当前仅单场景可练习，无法满足“四场景循环练习”目标。
- 消灭最大风险：场景切换与判定几何若不同步，会出现“能切场景但不能正确结算”。
- 对应 first playable 阶段：把单场景可玩升级为多场景可重复练习。

## 3. 触达的闭环链路
- Trigger: 用户在 `READY/DONE` 选择场景并开始控制，随后结束会话。
- Input: `scenarioId`、控制输入（方向/油门）、结束/重试/返回动作。
- Runtime/State: `selectedScenario`、`renderedScene`、`phase`、`ego`、`latchedRisks`。
- Core Logic: `scene-catalog -> scene-loader -> scene-renderer -> session-controller -> contract pipeline`。
- Derived Facts / Settlement: 场景几何差异、风险事实、成功/失败原因。
- UI Output: 练习页画布场景差异可见，结果面板给出对应场景的结算结果。

## 4. 影响的状态机
- 当前涉及状态：`IDLE`、`READY`、`RUNNING`、`SETTLING`、`DONE`。
- 新增/修改的状态迁移：
- `READY/DONE` 选择任一场景后重置到 `READY` 并加载该场景。
- 首次有效控制触发 `READY -> RUNNING`。
- 手动结束或超时触发 `RUNNING -> SETTLING -> DONE`。
- `DONE -> READY`（重试当前场景）。
- `READY/DONE -> IDLE`（返回场景选择）。
- 不在本卡处理的状态：新增 phase、联网/存档相关状态。

## 5. 影响的数据流
- 输入数据：四场景配置、用户选择的 `scenarioId`、控制输入、结束动作。
- 运行时状态：`SCENARIOS`、场景几何规格、会话状态与车辆姿态。
- 派生事实：按场景计算的入位区域、碰撞区域、风险与最终结算原因。
- 展示状态：场景下拉选项、画布图层元素、HUD 与结果面板文案。

## 6. 范围（In Scope）
- 扩展四个固定场景并打通场景切换循环练习。
- 为四场景提供可区分几何渲染（正常/窄路、倒库/侧方）。
- 保证四场景都可完成完整交互结算。
- 更新 normal/boundary/failure/smoke 测试资产。

## 7. 非范围（Out of Scope）
- 不改车辆动力学模型与核心控制协议。
- 不做视觉主题重设计或美术资源引入。
- 不做历史回放、存档、联网能力。

## 8. 实现边界 / 约束
- 不允许修改 `auth/`、`infra/`。
- 不允许新增生产依赖。
- 公共接口如变更，必须补 contract test。
- 不引入临时 debug hook 到生产路径。
- JS/TS 改动后必须执行 `npm run lint && npm run test && npm run build`。

## 9. 建议改动模块
- `src/features/parking-contract/contract-constants.ts`：扩展 `SCENARIOS` 与场景契约常量。
- `src/scene-loader.ts`：提供四场景渲染规格装载。
- `src/scene-renderer.ts`：保证多场景图层输出一致且几何可区分。
- `src/features/parking-contract/result-classifier.ts`：补齐四场景入位区域判定。
- `src/features/parking-contract/risk-recorder.ts`：补齐四场景静态障碍/风险几何。
- `src/play-page.ts` 与 `src/web-app.js`：切场景循环练习交互接线。
- `index.html`：四场景入口选项与文案对齐。
- `tests/parking-contract/normal-path.spec.ts`、`tests/parking-contract/boundary-path.spec.ts`、`tests/parking-contract/failure-path.spec.ts`、`tests/parking-contract/smoke-checklist.md`：覆盖四场景闭环。

必须联动层：
- 场景契约层（constants/loader/renderer）必须同改。
- 判定与风险层（classifier/risk-recorder/session-end-policy）必须同改。
- 交互层与测试层必须同改，保证“可见即可测”。

## 10. 验收标准（行为式）
1. Given 在 `READY`，When 用户选择任一场景并加载，Then 画布展示该场景且 `renderReady=true`。
2. Given 用户已完成场景 A 并处于 `DONE`，When 切换到场景 B，Then 会话回到 `READY` 且场景几何明显变化。
3. Given 用户依次切换四个场景完成一轮练习，When 每轮结束，Then 每轮都进入 `DONE` 并显示对应结果。
4. Given 非法 `scenarioId` 输入，When 调用选择场景，Then 当前已选场景与渲染结果保持不变。
5. Given 四场景任意一个，When 用户执行边界/失败路径，Then 结果面板能显示可解释失败原因（如速度超阈值/风险原因）。

## 11. 测试策略（有预算）
- 单元测试：场景 ID 校验、场景规格装载、场景化入位/碰撞几何判定。
- 状态机测试：`READY/DONE` 切场景后的状态重置与闭环迁移。
- 集成 smoke：
- “四场景循环：加载 -> 控制 -> 结束 -> 结果展示”。
- “跨场景边界/失败样例可复现并可解释”。
- 本卡不测：像素级视觉一致性与跨浏览器性能专项。

## 12. 实施步骤（建议 3~5 步）
- Step 1：扩展四场景契约常量与场景规格（渲染几何 + 判定几何）。
- Step 2：接通 loader/renderer 与会话切场景流程，确保 `READY/DONE` 可循环切换。
- Step 3：补齐四场景下 `result-classifier` 与 `risk-recorder` 的数据映射。
- Step 4：更新页面场景入口与 HUD/结果展示联动。
- Step 5：补齐 normal/boundary/failure/smoke 后执行 `lint/test/build` 全量门禁。

## 13. 风险与回滚
- 主要风险：场景渲染几何与判定几何映射不一致导致“看起来入位但判失败”。
- 回滚边界：仅回滚四场景扩展相关文件，不回滚 s01~s04 已稳定流程。
- 无破坏确认：回归 s01~s04 既有用例，再验证 s05 四场景新增用例。

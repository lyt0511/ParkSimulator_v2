# 任务卡 Slice s02 - 控制输入驱动的自车可视运动

## 1. 本卡要交付的用户能力
用户在 `READY/RUNNING` 状态下通过键盘或按钮输入控制后，主画布与 HUD 能实时显示“车头+车体+车轮”形态的自车运动与转向反馈。

## 2. 本卡为什么现在做
- 解决核心缺口：当前场景可见但“输入到车辆动态反馈”闭环不完整。
- 消灭最大风险：后续结算链路若缺少稳定的动态输入与姿态数据，将难以验证。
- 对应 first playable 阶段：从“看得见场景”推进到“可控可动且可观察”。

## 3. 触达的闭环链路
- Trigger: 用户在练习页触发键盘/按钮控制。
- Input: `direction + throttle` 控制输入。
- Runtime/State: `READY -> RUNNING`，并持续更新会话运行态。
- Core Logic: `input-dispatcher -> input-guard -> session-controller -> scene-renderer`。
- Derived Facts / Settlement: 形成连续的车辆位姿与可视层数据，结算仍用占位结果。
- UI Output: 主画布自车实时运动与 HUD 控制状态同步更新。

## 4. 影响的状态机
- 当前涉及状态：`READY`、`RUNNING`。
- 新增/修改的状态迁移：
- 有效输入触发 `READY -> RUNNING`。
- `RUNNING` 下连续控制形成自循环更新（不改变相位）。
- 非法输入不触发迁移，状态保持。
- 不在本卡处理的状态：`SETTLING`、`DONE` 的最终判定细化与风险策略。

## 5. 影响的数据流
- 输入数据：键盘与按钮控制事件（方向、动力）。
- 运行时状态：`lastControl`、自车位姿（x/y/angle）、`phase`。
- 派生事实：车头朝向、车轮位置、可见图层中的 `egoCar` 动态可视数据。
- 展示状态：HUD 实时控制信息、画布自车动态渲染结果。

## 6. 范围（In Scope）
- 接通键盘/按钮控制输入分发到统一输入链路。
- 复用并强化 `input-guard` 的方向校验与油门钳制。
- 在 `session-controller` 中维护自车位姿推进与相位切换。
- 在渲染层呈现“车头+车体+车轮”动态外观，并与 HUD 同步。

## 7. 非范围（Out of Scope）
- 不实现最终成功/失败分类规则升级。
- 不实现风险锁存、超时自动结束、压线/碰撞策略。
- 不扩展四场景差异渲染，仅保证当前场景输入闭环稳定。

## 8. 实现边界 / 约束
- 不允许修改 `auth/`、`infra/`。
- 不允许新增生产依赖。
- 允许最小必要的公开接口调整；若公共接口变化，必须补 contract test。
- 不引入临时 debug hook 到生产路径。
- 保持现有渲染层顺序契约不被破坏。

## 9. 建议改动模块
- `src/play-page.ts`：统一键盘/按钮输入入口，接线到会话控制。
- `src/features/parking-contract/input-guard.ts`：输入校验与边界钳制规则。
- `src/session-controller.ts`：`READY/RUNNING` 状态迁移与位姿推进。
- `src/scene-renderer.ts`：自车视觉规格映射与动态渲染输出结构。
- `src/web-app.js`：演示层按同一控制语义更新画布与 HUD。
- `tests/parking-contract/boundary-path.spec.ts`：边界输入与控制可见性验证。
- `tests/parking-contract/normal-path.spec.ts`：控制后进入 `RUNNING` 与可视反馈验证。

## 10. 验收标准（行为式）
1. Given 会话在 `READY` 且场景已渲染，When 用户输入一次有效控制，Then 会话转为 `RUNNING` 且自车在画布发生可见位姿变化。
2. Given 会话在 `RUNNING`，When 用户连续输入左/右/直行控制，Then HUD 的 `lastControl` 与自车朝向变化保持一致。
3. Given 输入 `throttle=1.3` 或 `-1.7`，When 输入被处理，Then 实际动力值分别钳制为 `1` 与 `-1` 且会话不中断。
4. Given 输入非法方向值，When 输入被处理，Then 该输入被忽略且状态相位与上一有效控制结果保持不变。

## 11. 测试策略（有预算）
- 单元测试：`input-guard` 的方向合法性与油门钳制。
- 状态机测试：`READY -> RUNNING` 触发条件与 `RUNNING` 自循环更新。
- 集成 smoke：键盘和按钮两条输入链路均能驱动画布自车与 HUD 同步更新。
- 本卡不测：最终结算分类、风险锁存、超时结束策略。

## 12. 实施步骤（建议 3~5 步）
- Step 1：梳理并统一输入分发入口，保证按钮/键盘语义一致。
- Step 2：实现并接通输入归一化到会话控制推进。
- Step 3：补齐自车视觉渲染（车头/车体/车轮）与 HUD 同步展示。
- Step 4：补充并执行 normal/boundary 相关测试与最小 smoke 验证。

## 13. 风险与回滚
- 主要风险：演示层与契约层双实现可能出现输入语义漂移。
- 回滚边界：仅回滚输入接线、位姿推进与自车渲染相关文件，不触碰结算与风险模块。
- 无破坏确认：保持现有 `s01` 渲染可见性与既有占位结算行为测试通过。

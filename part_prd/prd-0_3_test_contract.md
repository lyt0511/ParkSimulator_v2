1. NP-01 正常倒库成功
   - 前置：选择“正常倒车入库”默认初始位姿。
   - 操作：执行无压线、无碰撞控制序列，在 300s 内手动结束。
   - 期望：入位成功=是；安全合规=是；会话状态=已完成。

2. NP-02 正常侧方成功
   - 前置：选择“正常侧方停车”默认初始位姿。
   - 操作：允许中途多次前后修正，全程无压线/碰撞，在 300s 内手动结束。
   - 期望：入位成功=是；安全合规=是；会话状态=已完成。

3. NP-03 窄路倒库成功
   - 前置：选择“窄路倒车入库”默认初始位姿。
   - 操作：执行较长控制序列但不触发风险，在 300s 内手动结束。
   - 期望：入位成功=是；安全合规=是；会话状态=已完成。

## 2. 边界路径（3 条）
1. BP-01 速度阈值边界
   - 操作：仅改变结束瞬时速度，分别测试 0.05m/s 与 0.051m/s。
   - 期望：0.05m/s 可判入位成功；0.051m/s 不可判入位成功。

2. BP-02 时间阈值边界
   - 操作：299.9s 手动结束；另一轮达到 300.0s 仍不手动结束。
   - 期望：前者输出正常最终判定；后者自动结束并标记未完成（超时）。

3. BP-03 输入边界
   - 操作：输入动力 1.3 与 -1.7，方向输入非法值。
   - 期望：动力分别被约束到 1.0 与 -1.0；非法方向不进入有效控制；会话不中断。

## 3. 失败路径（3 条）
1. FP-01 碰撞后入位
   - 操作：过程中发生碰撞，结束时满足入位与静止条件。
   - 期望：入位成功=是；安全合规=否；风险标记包含碰撞。

2. FP-02 压线后入位
   - 操作：过程中发生压线，结束时满足入位与静止条件。
   - 期望：入位成功=是；安全合规=否；风险标记包含压线。

3. FP-03 超时未完成
   - 操作：300s 内不手动结束。
   - 期望：会话自动结束；入位成功=否；安全合规=否；会话状态=未完成（超时）。

## 4. 验收标准（必须可观察、可测试）
1. 四个场景均可启动并完成一次会话，且每次都有实时状态与最终结果展示。
2. 最终结果必须同时展示两个字段：入位成功（是/否）与安全合规（是/否）。
3. 压线或碰撞一旦发生，最终结果中必须保留对应风险标记。
4. 会话结束规则可验证：手动结束立即出结果；300s 未手动结束自动结束并标记超时。
5. 输入约束可验证：动力超范围被钳制，非法方向输入不生效。
6. 阈值判定可验证：速度 0.05m/s 与 0.051m/s 的判定结果必须不同且符合规则。
7. 性能可验证：状态刷新频率 >= 20Hz；控制到状态变化 < 100ms；FPS >= 30（桌面 Chrome 稳定版）。
8. 稳定性可验证：连续运行 30 分钟无崩溃、无会话卡死。
9. 一致性可验证：同一输入序列重复运行，最终判定一致。
10. 准确性可验证：40 条预设用例（每场景 10 条）中，系统最终判定与人工标注一致率 >= 90%。
11. 输出边界可验证：不产生文件输出，结果仅在前端界面可见。

## 5. Definition of Done
1. 正常/边界/失败 9 条路径全部执行并通过。
2. 验收标准 1-11 条全部通过并有记录。
3. 40 条基准用例一致率达到 >= 90%。
4. 关键规则文档化并一致：双结果输出、风险不硬中断、300s 超时、速度阈值判定。
5. 无阻断级需求缺口（无法判定或无法验收的问题）。

## 6. 仍存在的需求风险
1. 人工标注存在主观性，可能影响“>=90% 一致率”的可信度。
2. 速度与边界容差的体感合理性未验证，可能出现“规则通过但用户不认可”。
3. 当前验收仅覆盖桌面 Chrome，跨浏览器行为风险仍在。
4. 无历史记录与回放，异常复盘和定位效率受限。
5. 简化模型与真实驾驶存在偏差，训练迁移效果有不确定性。
6. 当前不提供动作建议，用户可能只能看到结果而难以改进策略。

## 7. 关键实现细化（补充契约）
### 7.1 顶层状态机
#### 7.1.1 状态定义
1. `IDLE`：未进入会话，未选择场景。
2. `READY`：已选场景并加载默认初始位姿，等待开始。
3. `RUNNING`：会话进行中，逐帧处理输入、车辆推进、风险评估与 HUD 刷新。
4. `SETTLING`：已触发结束，冻结有效输入，执行最终判定。
5. `DONE`：最终结果已生成并展示（包含双结果与风险标记）。

#### 7.1.2 状态转移与触发方
1. `IDLE -> READY`
   - 触发：用户选择场景。
   - 发起方：UI。
   - 执行方：`session-controller`。
2. `READY -> RUNNING`
   - 触发：用户点击开始，或首次有效控制输入到达。
   - 发起方：UI / 输入事件。
   - 执行方：`session-controller`。
3. `RUNNING -> SETTLING`（手动结束）
   - 触发：用户点击“完成/判定”。
   - 发起方：UI。
   - 执行方：`session-end-policy`。
4. `RUNNING -> SETTLING`（超时结束）
   - 触发：会话计时达到 `300.0s` 且未手动结束。
   - 发起方：系统计时器。
   - 执行方：`session-end-policy`。
5. `SETTLING -> DONE`
   - 触发：最终分类完成。
   - 发起方：`result-classifier`。
   - 执行方：`session-controller`。
6. `DONE -> READY`
   - 触发：用户点击重试当前场景。
   - 发起方：UI。
   - 执行方：`session-controller`。
7. `DONE -> IDLE`
   - 触发：用户返回场景选择。
   - 发起方：UI。
   - 执行方：`session-controller`。

#### 7.1.3 非转移事件约束
1. 压线/碰撞/出界仅记录风险，不在 `RUNNING` 期硬中断状态机。
2. 结束优先级：同帧同时出现“手动结束信号”和“300s 超时”时，按手动结束处理。

### 7.2 单帧数据流（Frame Tick）
1. 输入进入：
   - 控制输入先进入 `input-dispatcher`（方向离散，动力连续）。
2. 归一化：
   - `input-guard` 执行动力钳制 `[-1, 1]`，非法方向输入丢弃。
3. 车辆状态推进：
   - `session-controller` 读取归一化输入和 `dt`，推进车辆运动学状态。
4. 几何评估：
   - 几何评估器计算 `coverage`、`angleDeg`、`outOfBounds`。
   - `risk-recorder` 锁存 `collision`、压线等风险事实。
5. 结算触发：
   - `session-end-policy` 每帧检查手动结束与超时条件；
   - 条件满足即进入 `SETTLING`，由 `result-classifier` 生成最终双结果。
6. UI 更新：
   - HUD 在 `RUNNING` 每帧刷新实时状态；
   - 结果面板在 `DONE` 一次性更新：入位成功/安全合规/风险标记/失败原因。

### 7.3 数据分层表（4 类）
| 数据层 | 字段示例 | 主写入者 | 更新频率 | 主要消费者 |
| --- | --- | --- | --- | --- |
| 配置数据 | 场景 schema、车位几何、阈值（`speed=0.05m/s`、`timeout=300s`） | `contract-constants` + 场景装载器 | 会话启动或切场景时 | 输入约束、几何评估、结算策略 |
| 运行时状态 | `phase`、车辆状态、选中场景、计时器 | `session-controller` | 每帧 | HUD、结算策略、结果分类 |
| 派生事实 | `coverage`、`angleDeg`、`stillSeconds`、`collision`、`outOfBounds` | 几何评估器 + `risk-recorder` | 每帧（风险锁存可跨帧保留） | HUD、`result-classifier` |
| 展示状态 | HUD 文案、结果面板、失败原因文案 | UI view-model 组装层 | HUD 每帧；结果面板结算后 | 前端展示组件 |

补充约束：
1. 只允许“配置 -> 运行时 -> 派生 -> 展示”的单向读取依赖。
2. 展示状态不得直接回写运行时状态与派生事实。

### 7.4 最小可玩定义（MVP）
用户可在 4 个固定场景中任选其一完成一轮停车尝试，并在手动结束或 300 秒超时后看到实时状态与最终双结果（入位成功/安全合规及风险原因），形成完整可测试闭环。
